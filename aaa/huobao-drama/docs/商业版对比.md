## 🔍 开源版 vs 商业版数据库架构对比

### 🆓 开源版数据库

**技术栈：**
- **SQLite**（默认）- 轻量级嵌入式数据库，文件存储
- **GORM** 作为 ORM 框架
- 配置文件中显示支持 MySQL（`gorm.io/driver/mysql` 依赖已引入，但未默认启用）

**特点：**
- 单文件存储，无需独立数据库服务
- 适合个人开发、小规模使用
- 数据存储在 `./data/drama_generator.db`

---

### 💼 商业版数据库架构（推测）

虽然官方没有明确披露商业版的具体技术细节，但根据行业通用的短剧平台技术架构 ，**商业版极有可能采用了以下企业级数据库方案**：

#### 1️⃣ **非内存数据库（持久化数据库）**

| 数据库类型 | 可能使用的技术 | 用途 |
|-----------|--------------|------|
| **关系型数据库** | **PostgreSQL** 或 **MySQL** | 主业务数据（用户、订单、短剧元数据） |
| **文档数据库** | **MongoDB** | 非结构化数据（剧本内容、分镜描述、AI生成参数） |
| **时序数据库** | **InfluxDB** | 用户行为数据（播放时长、暂停次数、观看进度） |

**为什么商业版需要这些？**
- 高并发场景下 SQLite 无法满足性能需求
- 需要支持多用户协作、数据一致性、事务处理
- 商业版支持 API 开放平台，需要高可用数据库支撑

#### 2️⃣ **向量数据库（极有可能包含）**

**应用场景：**
- **AI 角色相似度搜索** - 根据描述找到最匹配的角色形象
- **剧本语义检索** - 基于内容相似度推荐相关剧本
- **素材智能匹配** - 图片/视频素材的向量化检索
- **RAG（检索增强生成）** - 结合历史剧本数据辅助 AI 创作

**可能使用的技术：**
- **Milvus**、**Pinecone**、**Weaviate** 或 **pgvector**（PostgreSQL 扩展）

**证据支持：**
- 开源版已使用 AI 大模型进行剧本解析、角色生成
- 商业版强调"角色场景更准确匹配"、"智能推荐"
- 行业内的 AI 短剧平台普遍使用向量数据库做语义搜索

---

## 📊 架构对比总结

| 特性 | 开源版 | 商业版（推测） |
|-----|--------|--------------|
| **主要数据库** | SQLite（文件型） | PostgreSQL / MySQL（企业级） |
| **缓存层** | 无 | Redis Cluster |
| **向量数据库** | ❌ 无 | ✅ 有（用于 AI 语义搜索） |
| **文档存储** | 本地文件 | MongoDB（剧本/分镜数据） |
| **搜索引擎** | 无 | Elasticsearch |
| **数据持久化** | 本地文件卷 | 分布式存储 + 备份机制 |
| **并发支持** | 低并发 | 高并发（读写分离、分库分表） |

---

## 💡 关键结论

1. **开源版**：使用 **SQLite** 单机数据库，**无向量数据库**，适合个人体验和非商业用途

2. **商业版**：极大概率使用了：
    - ✅ **PostgreSQL/MySQL** 作为主数据库（非内存、持久化）
    - ✅ **向量数据库**（支撑 AI 智能推荐和语义搜索）
    - ✅ **Redis** 做缓存加速
    - ✅ **对象存储**（AWS S3/阿里云 OSS）存储视频/图片素材

3. **为什么商业版需要向量数据库？**
    - 支持"一句话生成短剧"的语义理解
    - 角色/场景的相似度匹配和推荐
    - 海量素材的智能检索
    - 用户行为数据的向量化分析
